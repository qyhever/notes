## 演变历程

### 单机架构
故事的起点很简单，你有一个网站，用户量很少，访问量极低，
这时候怎么简单怎么来，一台服务器上安装着应用程序和数据库，用一套开源的LAMP/LNMP（Linux、Apache/Nginx、MySQL、PHP）技术栈就可以迅速搞定。

### 分机部署/垂直拆分
业务慢慢有了起色，用户多了一点，单机扛不住，应用和数据库经常抢资源怎么办？很简单嘛，加机器，把应用和数据库分开，一台机器跑应用专门处理用户请求和业务逻辑，一台机器跑数据库专门存数据做IO操作，两者各司其职，互不干扰，可用性也提升了。

### 集群
但好景不长，用户又暴涨了，单台应用服务器处理不过来，数据库也扛不住大量查询了，这里高并发来了。单台应用服务器就算配置拉满也有上限，比如一秒只能扛1000个请求，来了5000个请求怎么办呢？5000/1000=5嘛，再来5台服务器，部署一模一样的代码。

### 负载均衡
还有个问题，流量怎么分配到每台服务器呢？于是你发明了一个中间层，它能将请求均匀的分配到每台应用服务器上，这就是负载均衡器（Load Balancer）。应用服务器扛不住就加机器，取名叫水平扩展。一台挂了可以自动屏蔽流量，其它的来顶上，保证系统不会崩，取名叫高可用。

应用扛住了，新的问题来了，大量请求穿透到数据库，数据库成了新瓶颈。又在数据库前面加了个缓存层，比如Redis，把经常查的热点数据先存进去，查的时候先看缓存有没有，有就直接返回，没有再查数据库，这样数据库压力就小多了。设计了三层缓存，浏览器缓存应用服务器、本地缓存、分布式缓存。

### 读写分离/主从复制
缓存解决了热点读，但写请求和非热点读还是要打到数据库怎么办呢？这时候你发现数据库的写入和查询压力还是很大，于是你决定把数据库也拆开，一台专门负责写入，一台专门负责查询，这就是读写分离。

数据库有个特点，写比读慢的多，写会锁行锁表，并发一高就排队。

主库只负责写，从库只负责读，主从同步，主库写完数据，从库实时同步。

现实中系统读多写少，所以可以一主多从，主库写，从库读，这样读的压力就分摊了。

### 分库分表
跑了几年，数据量越来越大，订单表日志表越来越大，单库单表撑不住了，这时候必须对数据库进行拆分了。

按业务拆分，用户库商品库订单库互不干扰，取名叫垂直分库。

一张大表拆成多张小表，比如按UID哈希分成多张表，分散压力，配合数据库中间件，对应用透明，叫水平分表。

到这里数据库层面从单点变成分布式，存储和并发能力可以无限扩展了。

### CDN
后来越做越大，用户遍布全国各地，有人访问快，有人访问慢的像蜗牛。

于是接入了CDN，将静态资源缓存到离用户最近的节点，访问速度直接起飞。

### 反向代理
而且应用服务器直接暴露在公网，安全问题怎么解决呢？
用户和应用中间又加了一层反向代理，公网请求先到它，再转发给内网应用服务器。它隐藏了真实应用服务器，可以防攻击，还能做流量清洗。

网关相较于反向代理，属于更上层的服务治理组件，除了请求转发，还附带鉴权、限流、路由、日志、协议转换等功能（如 Spring Cloud Gateway、Kong）。
API 网关本质也是一种增强版的反向代理。

### 搜索引擎和NoSQL数据库
传统数据库遇到模糊查询、全文搜索、多维统计有点力不从心了。比如电商搜商品，新闻热搜用like太慢。

一个利用倒排索引让搜索秒级响应，取名叫搜索引擎(Elasticsearch)。

一个专门存非结构化半结构化数据，而且支持高并发易扩展，取名叫NoSQL数据库(MongoDB)。

### 分布式架构
生意越做越大，应用侧又有新问题了，用户商品订单 所有功能代码全放在一个应用里，变成了巨石应用。

改一行用户相关的代码，要全量发布整个应用。
多个开发同时改代码，天天冲突不断。
想给订单服务加机器扛流量，只能整个应用一起扩容，大把资源白白浪费。

按照业务边界，将庞大的应用拆分，独立用户系统，独立商品系统，独立订单系统，每个系统单独部署，单独扩容，互不干扰。

### RPC
应用拆分后，问题又来了，用户下单的时候要查商品库存，生成订单，这些系统都分开了，相互之间怎么调用，怎么通信呢？使用RPC，它让跨机器的服务调用像调用本地代码一样简单。

### 服务注册与发现中心(Nacos)
但是服务多了，相互调用的链路越来越复杂， 服务之间调用链路变长，一个请求可能要跨好几个服务，快速找到目标去调用成了难题，出问题了很难定位。而且服务之间调用失败了怎么办？怎么保证数据一致性？怎么监控和追踪请求？

于是发明了一个总管，它管着所有服务的地址，新增的服务都到它这里来注册，谁要调用谁，直接来这里找就行。

例如 Nacos、Eureka。

### 消息队列
服务直接相互调用，相互等待，一个服务卡住，整条链路都跟着堵死。能不能让服务不在等待呢，于是你接入了消息队列，服务直接不用干等着对方立刻回复，把消息扔进去就可以继续干自己的事，等对方处理完再回消息。

例如 RabbitMQ、RocketMQ、Kafka

### 微服务
分布式架构用了几年，发现拆是拆了，但是还不够细。例如用户系统，里面又有用户登录、用户信息、用户权限、用户积分、用户会员等业务，这些业务又可以继续拆分成更小的服务。每个服务可以用自己的技术栈，单独部署，单独扩容，互不干扰。

服务一多，调用关系像蜘蛛网，出问题找不到在哪，怎么办呢？于是你引入了分布式链路追踪，每个请求都有个唯一ID，从进入网关开始，记录下每个服务的处理时间、参数、返回结果，最后生成一张完整的调用图，哪里慢、哪里挂、一目了然。

全链路追踪: 从用户点击页面开始，从头到尾整条请求完整路径，包含前端 + 后端 + 中间件 + 数据库。

流量太大，怕被冲垮，怎么办呢？于是你引入了限流、降级、熔断机制，就像给系统装上保险丝，某个服务扛不住了，及时切断，避免雪崩。

服务太多不好管，于是你引入了服务治理，统一管理服务的注册、发现、配置、监控，让整个系统更加稳定可靠。

微服务架构真好用，从最初一个应用分成了几百个微服务，部署运维直接变成噩梦，于是你引入了容器化，比如 Docker，把服务打包成镜像，随时随地部署，快速回滚。

几百上千个容器，怎么管理呢？于是你引入了编排工具，比如 Kubernetes，它像一个看不见的管家，自动处理容器的部署、扩缩容、故障恢复，让你专注于业务逻辑。

就算有了 K8S，还是得自己买服务器，租机房，为了大促备了一堆机器，平时全闲着，成本高得要命。于是你直接把系统搬到了云平台，云平台就像一个无限大的资源池，需要多少就开多少，用完就关，按需付费，成本低得惊人。

比如 AWS、Azure、阿里云，它们提供虚拟机、存储、网络等基础设施，让你专注于业务逻辑。

最后，你的架构测底进入了新的时代，取名叫云原生。

## 总结
单机架构 -> 集群架构 -> 分布式架构 -> 微服务架构 -> 云原生架构

单机架构：一个应用运行在一台服务器上  
集群架构：多个应用运行在多台服务器上  
分布式架构：多个应用运行在多台服务器上，通过消息队列进行通信  
微服务架构：多个微服务运行在多台服务器上，通过API进行通信  
云原生架构：多个微服务运行在云平台上，通过容器进行部署  

单机架构：从零搭建  
应用与数据分离：资源争抢  
应用集群+负载均衡：高并发  
多级缓存：查询性能、数据库压力  
数据库读写分离：数据库读写阻塞  
分库分表+分布式数据库：海量数据存储  
CDN+反向代理：访问速度、系统安全  
搜索引擎+NoSQL：复杂查询  
业务拆分+分布式：业务膨胀、代码维护  
微服务架构：团队协作、极致弹性  
容器化+云平台：运维、成本、自动化
